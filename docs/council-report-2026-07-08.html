LLM Council · Verdict

How to build a UI for cheap_flights

5 advisors · anonymized peer review · chairman synthesis — 2026-07-08
The question. The backend is not a REST API — it's an LLM agent workflow (an Opus playbook run inside Claude Code). Only the Google Flights scraper (scripts/search.js) is callable without a model; Kiwi + lastminute — the only sources with one-click booking links — are MCP tools and were assumed unavailable headless. Hard rules: ≥2 sources, travel-time ceiling, rank on effective fare, every option a booking URL, ≥5 options. Four options weighed: A drive+stream the agent · B thin API over the GF scraper only · C rebuild a normal backend · D viewer over prior results.

Where each advisor landed

Contrarian
A/B/C all misframe ≥2-source as a UI problem — it's runtime availability. B is "a UI that lies." Prove MCP works headless before writing a line.
First Principles
A/B/C/D is a false choice. Real variable: must search be synchronous with viewing? No. A + D are one async pipeline. Viewer first, trigger second. Skip B & C.
Expansionist
The reasoning stream is the product — a "reasoning theater" Kayak can't copy. Render ROUTES.md as a living graph. Start with A, stream as first-class UI.
Outsider
Won't wait 60s or pay to search. But needs a real Book button. Fast GF result now, "get real bookable prices" behind a button that triggers the agent.
Executor
Only B + D is buildable this session, headless & model-free. Ship the viewer in <60 min, wire a Search button, surface the GF-only/no-deeplink gaps honestly.

Where the council agrees

  • Search doesn't have to be synchronous. Nobody books a flight in 3 seconds — the pain (cost, latency, auth, non-determinism) all comes from making a human wait on the agent. (First Principles, Outsider, Executor)
  • The ≥2-source and booking-link rules are load-bearing — a UI that quietly drops them "lies." (Contrarian, First Principles, Outsider)
  • C is the wrong move — rebuilding a conventional backend duplicates the one thing the project already is. (all five)
  • The file-reading viewer surface is honest and buildable today against real .captures/*.json. (Executor, First Principles)

Where the council clashes

  • Speed vs. depth. The Outsider wants an instant, free first result; the Expansionist says the slowness is the product and instant tables are commodity. Resolution: show the fast result and the deep one — fast GF paint first, agent depth behind an explicit action.
  • Does B have any legitimate role? The Executor builds on B+D as the only session-shippable thing; the Contrarian calls B "the trap." Resolution: B is acceptable as a labeled, partial first view — never as the shipped product presented as complete.
  • Start minimal (viewer) or architect for the full stream? First Principles/Executor say viewer-first; Expansionist says design the reasoning-stream cockpit from day one.

Blind spot the council caught (unanimous)

All 5 reviewers independently picked the same strongest response (First Principles) and the same biggest blind spot (Expansionist) — and all 5 flagged the same missing move.
  • The entire debate rests on an untested assumption. Every advisor treated "Kiwi/lastminute don't work headless" as fact. It isn't — those MCP tools are callable in this very environment. One probe call answers the question that decides A vs. B vs. renegotiating the rules. The council theorized over something it could have measured.
  • The Expansionist sold the theater without checking the engine starts — streaming a run that can't produce booking links or a 2nd source is a glass cockpit with no bookable flights.
  • Nobody asked whether the hard rules themselves are renegotiable product requirements vs. inherited constraints (Reviewer 4).

The recommendation

Chairman's synthesis

Adopt the async architecture (First Principles): decouple finding the truth from acting on it. Build the results-viewer surface first — it reads .captures/*.json and is honest against real data today. Wire an async agent trigger second: the UI submits a route, the agent run writes files (with booking links), the viewer reads them.

Fold in the Outsider's UX truth: paint the fast Google-Flights result immediately, clearly labeled "1 source, prices only", with a "Get real bookable prices" action that kicks the full agent run. This fuses B's fast first-paint with A's depth — satisfying the correctness rules by disclosure, not by pretending. Keep the Expansionist's reasoning-stream + living ROUTES.md graph as the phase-2 north star. Skip C.

But the architecture is gated on one fact the council never checked — whether the MCP booking sources actually work in an automated run. Measure it before committing.

The one thing to do first

Run the probe: call the Kiwi search-flight and lastminute generate_booking_link MCP tools once, right now, and confirm whether they return results + a real booking URL in this environment. That single fact decides how much of the plan is buildable — build the viewer immediately after.

Full advisor responses

Contrarian The ≥2-source rule is a runtime problem, not a UI one

The framing hides the real fault line: A, B, and C all pretend the ≥2-source rule is a UI problem when it's actually a runtime-availability problem. Kiwi and lastminute are MCP servers callable only through a model tool-call, and absent in headless runs. That single fact quietly kills B and D, cripples C, and makes A far worse than advertised.

B is the trap. "Fast, cheap, headless" — and it structurally cannot satisfy your own hard rules. One source, no booking URL, no cross-check. You'd ship a UI that violates every correctness invariant the project exists to enforce. That's the "beautiful UI that lies."

A's hidden killer is auth, not speed. If MCP can't authenticate non-interactively, even Option A can't reliably produce booking links or a second source. Has anyone confirmed MCP works outside an interactive Claude Code session? That's the question everyone's avoiding.

C duplicates the agent yet has no legal 2nd-source API to reimplement. D isn't a strategy, it's a screenshot. Before writing one line: prove MCP can be driven programmatically and produce a booking link headlessly. If it can't, your only honest options are A (accept slow/costly/interactive) or explicitly renegotiate the ≥2-source rule.

First Principles Must search be synchronous with viewing? No.

You're treating "the UI" as one thing. It isn't, and that's the whole confusion.

A flight search has two jobs welded together: Job 1 — find the truth (cross-check, rank effective fare, verify — the expensive, non-deterministic agent work, already built). Job 2 — let a human act on it (browse, click a booking link — cheap, deterministic, where the impeccable-UI value lives).

So the real question isn't A/B/C/D. It's: must search be synchronous with viewing? Everything painful comes from forcing a human to wait while an Opus agent thinks. Nobody books a flight in the 3 seconds an API returns.

That collapses the false choice. A and D aren't rivals — they're two ends of one async pipeline: the UI submits a route (kicks an agent run), the run writes .captures/*.json + booking links to files, and the UI reads them. B and C exist only to fake synchrony, and each breaks a hard rule to do it.

Build the viewer first (D) — honest today against real captures. Wire the async trigger (A) second. Skip B and C. The user wants a right answer, not a fast one.

Expansionist This isn't a flight site — it's a reasoning theater

Every flight site shows a sorted table. None show the thinking. Option A's "slow and non-deterministic" is the product, not the bug. Watching Opus expand "cheap Lisbon in October" into gateway nodes, fire five sub-agents, and catch a fare that's a mirage on effective cost — that's a moat Kayak structurally cannot copy. Trust compounds when you see the work.

ROUTES.md is the sleeper asset. Render it as a live route-topology graph — gateway nodes brightening with confidence, decaying edges fading. Every search enriches a shared brain. A data flywheel hiding inside a markdown file. Expose it.

The 10x version: not a search box but a glass-cockpit trip planner where the reasoning stream is the interface — expandable plan cards, live sub-agent status, an interactive ROUTES.md graph, one-click booking links the instant judgment lands, the user steering mid-flight.

Start with A, architect the stream as first-class UI. B and D are demos of a corpse; C rebuilds the one differentiated thing. Ship the reasoning.

Outsider I don't want to watch a robot think — I want a plane ticket

Option A sounds cool for ten seconds, then I'd hate it. Sixty seconds is an eternity — I abandon sites that take four. And it costs money to search? I'm gone. But A is the only one with real Book buttons; a result I can't book is a screenshot.

Option B feels great and is broken as a product: instant and free, but no booking links = a fast dead end. Option C is a long project — not the question. Option D is a museum.

What would feel best: the speed of B with the booking links of A. Start with the fast free scraper for an instant result, show options immediately, and put a "Get real bookable prices" button that triggers the slow/expensive agent only when I've decided I care. Ship B's frontend now, wired so A plugs in behind that button.

Executor B + D fused — the only thing buildable this session

Build B + D fused into one app — the only thing that runs headless, model-free, and demoable this session. A and C are dead on arrival: A needs interactive MCP auth you don't have; C is a rewrite.

Order: (1) Read-only harness — server.js shells out to node /workspaces/other_project/scripts/search.js, reads the capture JSON, serves it; never write into the read-only backend. (2) Ship the viewer first — point the frontend at an existing gflights_results.json → a real ranked table in <60 min. (3) Wire a Search button → POST → repaint = live, deterministic, GF-only.

No-display reality: never headed; drive/verify with headless Playwright + screenshots/trace. Surface gaps honestly: GF-only → show "Kiwi/lastminute unavailable headless"; no deeplink → link to GF search, mark it. Booking links + multi-source wait for interactive mode.
